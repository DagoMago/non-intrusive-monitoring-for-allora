One implementation detail that is worth discussing is the separation between the execution of the sensing functions and that of the control/metrics transmission functions. In other words, it is useful to illustrate how one of the main objectives of this project was guaranteed: ensuring that the management of control commands and communications does not interfere with the normal functions of the end nodes, namely metric generation and all the sensing-related functionality involved.

Firstly, in the case of the LilyGO sensor nodes, this guarantee is implemented in the main Arduino program by assigning the periodic metric generation task to the second core (core 1). Specifically, the execution of the initialization function (setup) creates a task explicitly pinned to that core. The rest of the code, associated with control command handling and communication management, remains implicitly assigned to core 0.

Secondly, the Raspberry Pi sensor nodes follow a similar approach in their main MicroPython program. When the node starts, it creates a new execution thread dedicated to the periodic generation of metrics, since that thread is responsible for running the update_metrics_loop function. Meanwhile, the main thread executes the logic that distinguishes among the different control commands that may have been received and runs the code associated with each of them according to the detected command, in addition to the transmission of metrics itself.

It is worth highlighting a distinguishing aspect between both execution schemes, beyond the use of execution threads in one case and core separation in the other. And that aspect is that when comparing the two serial protocols used in the project, I2C is more appropriate in the particular case of this development. Its communication model gives full control to the node with the role of master, so that whenever that master wishes to obtain information provided by the slave, it simply reads from the serial channel. As a result, a read operation by the master triggers the slave, which generates the appropriate data and places it on the serial channel. This is better suited than UART to the metric transmission scheme, or control command scheme, that is intended here. With I2C, the code can be designed so that when the controller node reads the I2C link, a function is executed on the controlled node that returns the desired metrics.

In contrast, in the case of UART, both interconnected devices communicate as equals, without one maintaining control over the other. Consequently, the I2C-style exchange cannot be replicated. That exchange remains transparent to almost the entire sensor node code, apart from the specific function that responds to read operations. Therefore, an alternative scheme must be used in the UART case, since it is not possible to simply generate the metrics periodically on the sensor node and read them from the controller node whenever convenient. Instead, an explicit command for their generation and transmission must be sent. Under this latter scheme, responses to periodic metric requests use the same logic as the handling of a specific control function. More precisely, the same logic used for the connection confirmation (C-ACK) is employed, considering that the latter already includes the instantaneous generation and transmission of metrics as an addition to the connection confirmation itself.

It should also be noted that the choice of Arduino for the I2C code was due to a limitation of MicroPython. Although most of the project code was developed in that language -because it is the one used by the official AlLoRa protocol code-, it does not provide the libraries required to implement an I2C slave. As a result, it was necessary to look for an alternative language that did provide those resources, and Arduino was ultimately chosen because of its simplicity and the prior familiarity with it.
